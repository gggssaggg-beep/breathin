# «Дыши» — план сборки и архитектура

Документ-компаньон к `ТЗ_приложение_дыхательных_практик.md`. Фиксирует технические решения, архитектуру, модель данных, дорожную карту и первый вертикальный срез. Ссылки вида «ТЗ §N» указывают на разделы техзадания.

---

## 1. Резюме и цель

Строим кроссплатформенное (iOS + Android) приложение дыхательных практик: пользователь выбирает технику из каталога (8 техник со счётом, метод Вима Хофа, 3 техники со свободным ритмом), настраивает длительности фаз и режим окончания (циклы или таймер), и практикует **с закрытыми глазами и выключенным экраном**. Вся обратная связь дублируется по независимым каналам: голос, звуковые сигналы фаз, метроном, вибрация, визуальная анимация.

Ключевой принцип и одновременно главный технический риск проекта — **точность тайминга фаз ±50 мс при выключенном экране и в фоне** (ТЗ §7, §11.1). Обычные таймеры (`Timer`, `Timer.periodic`, `Future.delayed`) этого не дают: на Android процесс троттлится Doze-режимом, на iOS приложение засыпает. Поэтому сердце архитектуры — **PhaseEngine с расписанием, привязанным к аудио-часам**: вся сессия компилируется в один непрерывный аудио-таймлайн, который проигрывается аудио-железом с сэмплной точностью, а все остальные каналы (вибрация, визуал) синхронизируются от позиции этого таймлайна как от единственного источника времени.

Порядок работ подчинён снятию рисков: первый вертикальный срез — «аудио-хребет» (квадратное дыхание 4-4-4-4 с выключенным экраном, ±50 мс, пауза по звонку), и только после его приёмки наращиваются каталог, настройки, визуал и соцфункции. Работа режется на мелкие партии по 3–4 единицы, каждая коммитится отдельно.

## 2. Обоснование стека и финальный список пакетов

### 2.1. Почему Flutter (решение принято, разворачиваем аргументацию)

- **Точный тайминг**: во Flutter доступен полный контроль над аудио-конвейером (ExoPlayer/AVPlayer через `just_audio`, C++-движок через `flutter_soloud`, свои `StreamAudioSource`). В React Native/Expo (ТЗ §8.2) точный метроном потребовал бы писать нативный модуль на обеих платформах — то есть фактически два нативных аудио-движка плюс JS-обвязка.
- **Фон и локскрин**: `audio_service` даёт готовый foreground service (Android) и интеграцию с `MPNowPlayingInfoCenter`/`UIBackgroundModes: audio` (iOS) — сессия живёт при выключенном экране «бесплатно», без самописной нативщины.
- **Размер**: релизный arm64 APK Flutter-приложения без тяжёлой графики — 10–14 МБ; с нашим аудио-бюджетом (~3–5 МБ синтезированных ассетов) укладываемся в 30 МБ (ТЗ §11.5).
- **Один рендер на обеих платформах** — дыхательные анимации (CustomPainter) ведут себя идентично на iOS и Android.

Правило зависимостей: сверху вниз. Доменный слой не знает про Flutter и плагины — движок тестируется как чистый Dart с фейковыми часами.

### 3.2. Поток данных сессии

```
Каталог техник + настройки пользователя
        │
        ▼
SessionPlanCompiler ──► SessionPlan (список событий с абсолютным t, мс)
        │                    события: phaseStart, voiceCue, metronomeTick,
        │                    countdownDigit, gong, sessionEnd
        ▼
AudioTimelineRenderer ──► непрерывный WAV-таймлайн всей сессии
        │                 (голос + сигналы фаз + метроном + гонг,
        │                  смиксованы в PCM с сэмплной точностью)
        ▼
just_audio + audio_service ──► АУДИО-ЖЕЛЕЗО = мастер-часы
        │
        │  player.position (опрос 20 Гц / Ticker 60 Гц)
        ▼
PhaseEngine.dispatcher ──► Stream<EngineEvent>
        ├──► HapticsService   (вибро-паттерн на смену фазы)
        ├──► Визуал сессии    (фигура, надпись фазы, отсчёт)
        └──► SessionController (номер цикла, прогресс, итог в drift)
```

### 3.3. PhaseEngine — сердце приложения

**Проблема.** `Timer.periodic`/`Future.delayed` исполняются event-loop'ом Dart: при выключенном экране Android загоняет процесс в Doze (таймеры задерживаются на секунды), iOS замораживает приложение целиком. Никакой «умный» таймер требования ±50 мс не выполнит.

**Решение — «часы = само аудио».** Наблюдение: для всех техник, кроме Вима Хофа, сессия **полностью детерминирована в момент старта** (длительности фаз × число циклов известны заранее; таймерный режим сводится к целому числу циклов — ТЗ §3.1). Значит, можно не «стрелять событиями по таймеру», а **скомпилировать всю сессию в один аудио-файл** и отдать его аудио-подсистеме ОС:

1. **SessionPlanCompiler** (чистый Dart): техника + настройки → `SessionPlan` — упорядоченный список `EngineEvent(t_ms, type, payload)`: подготовка «3…2…1», для каждого цикла — стартовый сигнал фазы, голосовая фраза, тики метронома (с акцентом на смене фазы), опциональный голосовой отсчёт секунд, в конце — гонг. Итоговая длительность известна точно.
2. **AudioTimelineRenderer**: миксует PCM-сэмплы событий в единый моно-поток 44 100 Гц / 16 бит. Событие с t=12 340 мс кладётся на сэмпл `round(12.340 × 44100)` — **точность размещения = 1/44100 с**. Между событиями — настоящие нулевые сэмплы (тишина), поэтому аудио-сессия «играет» непрерывно и ОС не усыпляет процесс.
   - Реализация в срезе 1: рендер во временный WAV-файл (10 мин ≈ 50 МБ на диске, terminable-temp — приемлемо).
   - Целевая реализация: `SessionAudioSource extends StreamAudioSource` — «виртуальный WAV»: заголовок + PCM генерируются лениво по запрошенным диапазонам байт (тишина = нули, сигналы копируются из декодированных ассетов). Ноль больших аллокаций, ноль диска, мгновенный старт.
3. **Воспроизведение**: `just_audio` внутри `audio_service`-хендлера. Аудио-железо тянет PCM своим кварцем — дрейф фаз **нулевой по построению**; критерий ±50 мс закрывается качеством самих сэмплов (атака сигнала ≤10 мс), а не таймерами.
4. **Один источник времени для всех каналов**: позиция `player.position` — единственные часы движка. Диспетчер (look-ahead-цикл) опрашивает позицию (на экране — `Ticker` 60 Гц, в фоне — `Timer.periodic` 50–100 мс) и эмитит `EngineEvent`, чей `t` попал в окно. Важно: от этого цикла зависят только **вибрация и визуал**, у которых допуск мягче (±100 мс не различимы тактильно/зрительно); строгий канал — аудио — в цикле не нуждается вообще. Даже если фоновый таймер притормозит, звук не сдвинется ни на сэмпл, а вибрация лишь чуть запоздает.
5. **Пауза/резюм**: `player.pause()` замораживает часы — все каналы останавливаются согласованно, потому что других часов нет. При резюме — `seek` к началу текущей фазы (UX-решение: возобновлять с целой фазы, а не с полуслова).

**Обработка прерываний (через `audio_session`):**

| Событие | Реакция |
|---|---|
| Входящий звонок / Siri / другой аудио-фокус (`interruptionEventStream`) | `pause()`, флаг `pausedByInterruption`; по окончании прерывания — авто-резюм с начала текущей фазы (критерий §11.3) |
| Выдернули наушники / отвалился Bluetooth (`becomingNoisyEventStream`) | `pause()` без авто-резюма (звук внезапно уйдёт в динамик — нежелательно) |
| Подключение BT-наушников | Продолжаем; известный сдвиг A2DP-латентности — см. риск R5 |
| Смерть процесса (OEM-киллер) | Foreground service `mediaPlayback` от `audio_service` защищает; сессия дополнительно чекпоинтится в drift каждые N циклов — при рестарте предлагаем «продолжить?» |

**Фон и экран:**
- Android: `audio_service` поднимает foreground service с типом `mediaPlayback` + уведомление с кнопками пауза/стоп — процесс не убивается Doze.
- iOS: `UIBackgroundModes: [audio]` + активная категория `.playback`; таймлайн содержит реальные (нулевые) сэмплы, значит система считает, что звук играет, и не усыпляет приложение.
- `wakelock_plus` включается **только** на экране сессии при включённом визуальном режиме (ТЗ §6.5); при выключенном экране он не нужен — жизнь обеспечивает аудио-сессия.

**Настройка `audio_session`:** категория `playback`, `mixWithOthers = false` (наша практика — основной звук), `duckOthers` не нужен; `androidAudioAttributes: usage=media`.

### 3.4. Режим Вима Хофа (ТЗ §2.3) — отдельный режим движка

Единственная техника с недетерминированной длительностью (задержка «до позыва»). PhaseEngine работает как **машина состояний раундов**, где детерминированные куски по-прежнему пре-рендерятся, а открытая задержка — единственное место с «живыми» часами:

```
WimHofRound (× rounds):
  1. breathing   — детерминирован: N дыханий × темп.
                   Пре-рендерится мини-таймлайн раунда: ритм-тики +
                   голосовой счёт («…28, 29, 30») + команда
                   «выдохни и задержи» в конце. Играется через just_audio.
  2. retention   — открытая фаза: секундомер ВВЕРХ.
                   Часы = Stopwatch, сверяемый с DateTime.now()
                   (допуск здесь секунды, не мс — риска тайминга нет).
                   Аудио-сессию держит живой тихий loop (тишина/фон)
                   через just_audio; отметки каждые 30 с / 1 мин —
                   one-shot через flutter_soloud.
                   Завершение: тап пользователя ИЛИ достижение цели.
  3. recovery    — детерминирован: «глубокий вдох, держи 15 с» —
                   пре-рендеренный клип с отсчётом.
```

Результаты `retention` каждого раунда собираются в `wimHofRetentions: [t1, t2, ...]` и пишутся в сессию (график прогресса — этап 2). Safety-контур: перед стартом — полноэкранное предупреждение (ТЗ §2.4), подтверждаемое явно.

### 3.5. Техники без счёта (ТЗ §2.2)

Полностью ложатся в общий конвейер: интро-инструкция (голос) + опциональный фоновый звук (loop через just_audio, отдельный слой) + периодические подсказки (Нади Шодхана: «левая… правая…» с настраиваемым интервалом — детерминированы, вшиваются в таймлайн) + гонг в конце. Отдельного режима движка не требуют — это тот же `SessionPlan`, просто без фазовой структуры.

## 4. Модель данных

### 4.1. Локальная БД (drift / SQLite)

Разворачиваем ТЗ §9. `User`, `Friendship`, `NotificationSetting` — сущности этапа 2, живут в Firestore (локально — только кэш). `Technique` — не таблица, а зашитый справочник (см. §5).

```sql
-- История практик (ТЗ: Session)
CREATE TABLE sessions (
  id                  TEXT PRIMARY KEY,          -- uuid
  technique_id        TEXT NOT NULL,             -- ссылка на каталог
  started_at          INTEGER NOT NULL,          -- epoch ms, UTC
  duration_sec        INTEGER NOT NULL,          -- фактическая
  cycles_completed    INTEGER NOT NULL,
  pattern_used        TEXT NOT NULL,             -- json: фазы/темп на момент старта
  wim_hof_retentions  TEXT,                      -- json: [сек по раундам], опц.
  completed           INTEGER NOT NULL DEFAULT 1,-- 0 = прервана
  shared              INTEGER NOT NULL DEFAULT 0,
  synced_at           INTEGER                    -- этап 2: отправлена в облако
);
CREATE INDEX idx_sessions_started ON sessions(started_at);

-- Последние настройки per-техника (ТЗ §3: «сохраняются per-техника»)
CREATE TABLE technique_settings (
  technique_id     TEXT PRIMARY KEY,
  end_mode         TEXT NOT NULL,      -- 'cycles' | 'timer'
  cycles           INTEGER,            -- 1..100
  timer_min        INTEGER,            -- 1..60 (для timer-техник 1..30)
  phase_secs       TEXT,               -- json: [4.0,4.0,4.0,4.0]
  tempo_multiplier REAL,               -- для 4-7-8: 0.5/0.75/1/1.25
  wim_hof_params   TEXT,               -- json: {breaths, paceSec, rounds, ...}
  feedback         TEXT NOT NULL,      -- json: {voice,sound,metronome,vibration,visual}
  sound_set_id     TEXT NOT NULL DEFAULT 'minimal',
  voice_id         TEXT,               -- 'ru_f' | 'ru_m' | 'en_f' | ...
  prep_seconds     INTEGER NOT NULL DEFAULT 3,
  count_seconds    INTEGER NOT NULL DEFAULT 0,  -- голосовой отсчёт секунд
  updated_at       INTEGER NOT NULL
);

-- Глобальные настройки (одна строка) — тема, язык, дисклеймер, приватность
CREATE TABLE app_settings (
  id                     INTEGER PRIMARY KEY CHECK (id = 1),
  theme                  TEXT NOT NULL DEFAULT 'system',
  locale                 TEXT,                    -- null = системная
  disclaimer_accepted_at INTEGER,                 -- критерий §11.6
  share_by_default       INTEGER NOT NULL DEFAULT 0,
  master_feedback        TEXT NOT NULL DEFAULT '{}'  -- глобальные дефолты каналов
);

-- Предупреждения «первый запуск техники с задержками» (критерий §11.6)
CREATE TABLE safety_acks (
  technique_id TEXT PRIMARY KEY,
  acked_at     INTEGER NOT NULL
);

-- Напоминания «пора подышать» (ТЗ §5)
CREATE TABLE reminders (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  time_hhmm TEXT NOT NULL,       -- '08:30' локальное
  days_mask INTEGER NOT NULL,    -- битовая маска пн..вс
  enabled   INTEGER NOT NULL DEFAULT 1
);
```

Статистика (календарь, суммы минут, streak) — SQL-агрегации по `sessions`, отдельных таблиц не требует. Streak считается запросом по дням с практикой (локальная TZ из `Intl`-системной, без сети).

## 5. Каталог техник как данные

Каталог — **данные-как-код**: `const`-список в `lib/domain/catalog/techniques.dart`. Не JSON-ассет: константы проверяются компилятором, тексты идут через ARB-локализацию, а справочник по ТЗ §9 «зашит в приложение».

### 5.1. Модель

```dart
enum TechniqueType { counted, timer, wimHof }
enum PhaseKind { inhale, holdIn, exhale, holdOut }
enum SafetyLevel { low, medium, high }
enum ScalingMode {
  perPhase,        // каждая фаза настраивается независимо
  ratioLock,       // слайдер базы, пропорция фиксирована (4-16-8)
  ratioOptional,   // perPhase + переключатель «держать пропорцию» (2-8, 2-10)
  tempoMultiplier, // фазы фиксированы, множитель темпа (4-7-8)
}

class Technique {
  final String id;
  final TechniqueType type;
  final String nameKey, descriptionKey, benefitKey, safetyKey;
  final SafetyLevel safetyLevel;
  final VisualShape visual;
  // counted:
  final List<PhaseSpec>? phases;
  final ScalingMode? scaling;
  final List<double>? tempoOptions;      // 4-7-8: [0.5, 0.75, 1.0, 1.25]
  final List<PhaseSpec>? simplifiedPhases; // 4-16-8: дефолтный режим 4-8-8
  final int defaultCycles;
  final int? recommendedMaxCyclesNovice; // 4-7-8 и 4-16-8: 4
  // timer:
  final int? defaultTimerMin;            // 5
  final (int, int)? timerRangeMin;       // (1, 30)
  final PeriodicCue? periodicCue;        // Нади Шодхана: «левая/правая», интервал
  final bool backgroundSoundOption;
  // wimHof:
  final WimHofDefaults? wimHof;          // breaths 30 (20–50), paceSec 2.0 (1.5–3.0),
                                         // rounds 3 (1–5), recoveryHoldSec 15,
                                         // retentionTargetSec? (опц. цель)
}

class PhaseSpec {
  final PhaseKind kind;
  final double defaultSec;
  final bool editable;
  final double min, max;   // шаг слайдера 0.5 с (ТЗ §3.2)
}
```

## 8. Первый вертикальный срез: «аудио-хребет»

### 8.1. Что входит

**«Квадратное дыхание 4-4-4-4 × 10 циклов, слышимое и точное при выключенном экране»** — минимальный путь через все слои:

- `SessionPlanCompiler` + каталог из одной техники (box);
- `AudioTimelineRenderer` (вариант с temp-WAV) + сигналы набора «Минимал» из Python-скрипта;
- `just_audio` + `audio_service` (foreground service / background audio) + `audio_session` (звонок, наушники);
- `HapticsService` (паттерны на смену фазы) как второй канал от тех же часов;
- минимальный `SessionScreen`: надпись фазы, отсчёт, номер цикла, пауза/стоп, wakelock. Без анимаций, без настроек, без БД (кроме заглушки записи сессии).

### 8.2. Почему именно он

1. **Снимает главный риск проекта первым.** Точность ±50 мс в фоне (критерий §11.1) — единственное требование, которое может заставить менять архитектуру или даже стек. Каталог, настройки, статистика и соцфункции — предсказуемая работа без неизвестных; тайминг — нет. Если гипотеза «часы = аудио-таймлайн» подтвердится на реальных устройствах, всё остальное — наращивание мяса на проверенный скелет.
2. **Проверяет сразу три критерия приёмки** (§11.1 точность, §11.3 звонок, косвенно §11.5 — вклад аудио-ассетов в размер) на самой дешёвой версии продукта.
3. **Валидирует пайплайн ассетов**: синтезированные тоны сразу слушаются «в бою» — рано выяснится, различимы ли фазы на слух с закрытыми глазами.
4. Голос сюда **не входит** сознательно: TTS-заглушка не влияет на архитектуру часов (голос — такой же сэмпл в таймлайне) и добавляется партией П8 без переделок.

### 8.3. Definition of Done (привязка к ТЗ §11)

- [ ] Сессия «квадрат 4-4-4-4, 10 циклов» проходит от старта до гонга: (а) экран включён; (б) экран выключен кнопкой питания; (в) приложение свёрнуто — на всех трёх сценариях звук непрерывен, сессия доходит до конца. **(§11.1, часть «с выключенным экраном, только по звуку»)**
- [ ] Инструментальная проверка ±50 мс: вывод устройства записывается (loopback-кабель или диктофон рядом), `tools/measure_drift.py` находит атаки сигналов и сравнивает интервалы с планом; максимальное отклонение ≤50 мс на фазу, без накопления дрейфа за 10 циклов. Прогон на 3 устройствах: Pixel (чистый Android), Samsung (One UI), Xiaomi (агрессивный киллер). **(§11.1, часть «±50 мс»)**
- [ ] Юнит-тесты `SessionPlanCompiler`: суммарная длительность плана, позиции событий, таймерный режим → целое число циклов с завершением по ближайшему концу цикла (ТЗ §3.1).
- [ ] Юнит-тест `AudioTimelineRenderer`: событие с t=X мс лежит на сэмпле round(X·44100/1000) (проверка по отрендеренному буферу).
- [ ] Входящий звонок в середине сессии: пауза; после звонка — возобновление с начала текущей фазы; счётчик циклов корректен. **(§11.3)**
- [ ] Выдёргивание проводных наушников → пауза (без авто-резюма). Bluetooth-наушники: сессия играет, не прерывается.
- [ ] Вибро-паттерны смены фаз срабатывают при выключенном экране; субъективная рассинхронизация со звуком не ощущается (динамик телефона).
- [ ] Пауза/стоп с экрана и с уведомления локскрина (audio_service-контролы) работают.
- [ ] Wakelock активен на экране сессии, снят после выхода.
- [ ] Код закоммичен партиями П1–П5 (см. §9), движок и рендерер покрыты тестами, `flutter analyze` чистый.

Если на каком-то устройстве DoD по фону провалится — см. план «Б» в риске R1 (решение принимается ДО партии П6).

## 10. Пайплайн аудио-ассетов: `tools/generate_audio.py`

Python + `numpy` + `scipy.io.wavfile`, без API-ключей и внешних источников (лицензионно чисто: 100 % синтез). Запуск: `python tools/generate_audio.py --out assets/audio` — детерминированный, ассеты воспроизводимы из кода.

### 10.1. Общие параметры

- **Формат: WAV PCM 44 100 Гц / 16 бит / моно** — сознательно НЕ ogg/aac: сигналы миксуются рендерером в PCM-таймлайн, WAV читается без декодера и без кодек-паддинга (у mp3/aac «плывёт» начало — прямой удар по ±50 мс). Итоговый вес набора ~1.5–2 МБ — в бюджет 30 МБ входит легко. Ogg остаётся для длинных фоновых лупов timer-техник (их играет just_audio напрямую, декодер его).
- Все сигналы: атака ≤10 мс (точность субъективного начала), релиз 100–150 мс (без щелчков), пик-нормализация к целевому dBFS.
- Единый sample rate ассетов = sample rate таймлайна → микс без ресемплинга.

### 10.2. Генерируемые файлы

| Файл | Синтез | Длит. | Уровень |
|---|---|---|---|
| `sets/minimal/inhale.wav` | восходящий свип 300→600 Гц, синус, экспон. глиссандо | 700 мс | −6 dBFS |
| `sets/minimal/hold_in.wav` | ровный тон 450 Гц | 500 мс | −6 dBFS |
| `sets/minimal/exhale.wav` | нисходящий свип 600→300 Гц | 700 мс | −6 dBFS |
| `sets/minimal/hold_out.wav` | ровный тон 400 Гц (чуть ниже — различимость двух задержек) | 500 мс | −6 dBFS |
| `sets/minimal/tick.wav` | клик: 1000 Гц, 30 мс, резкая экспон. огибающая | 30 мс | −14 dBFS |
| `sets/minimal/tick_accent.wav` | клик 1500 Гц, 40 мс (акцент смены фазы) | 40 мс | −10 dBFS |
| `common/prep_beep.wav` | короткий бип 800 Гц для отсчёта «3…2…1» | 120 мс | −8 dBFS |
| `common/gong.wav` | колокол: сумма негармонических парциалов f₀=220 Гц × [1, 2.0, 2.9, 4.2, 5.4], амплитуды 1/k, экспон. затухание τ≈1.5–2.5 с (у высоких — быстрее), лёгкая расстройка ±0.3 % для биений | 6 с | −3 dBFS |
| `common/wim_hof_pace.wav` | двухтактный тик «вдох-выдох» для ритма ВХ (этап 2) | 2×150 мс | −8 dBFS |

Скрипт также пишет `assets/audio/manifest.json` (id → файл, длительность, назначение) — рендерер и настройки читают наборы по манифесту, добавление набора «Чаша» на этапе 3 не трогает код.

### 10.3. Голос

- **MVP (заглушка):** `flutter_tts.synthesizeToFile` при первом запуске генерирует набор фраз («вдох», «задержка», «выдох», цифры 1–20, «приготовьтесь», инструкции timer-техник) в кэш устройства → рендерер вшивает их в таймлайн как обычные сэмплы. Латентность живого TTS таким образом обходится; риски формата — R4.
- **Позже (по ТЗ §7):** пакет предзаписанных фраз (нейро-TTS: ElevenLabs/Yandex SpeechKit, 2 голоса × 2 языка × ~40 фраз, ogg→конверсия в WAV на этапе сборки ассетов) кладётся в `assets/audio/voice/` — код не меняется, меняется источник сэмплов.

## 11. Риски и открытые вопросы

| # | Риск | Вероятность/удар | Митигация |
|---|---|---|---|
| **R1** | **Фоновый тайминг на Android-вендорах**: OEM-киллеры (Xiaomi/MIUI, Huawei, Samsung «спящие приложения») убивают даже foreground service `mediaPlayback` | средняя / критический | Снимается срезом №1 на матрице устройств (Pixel/Samsung/Xiaomi). Митигация: audio_service + непрерывный аудио-поток (не «тишина пауз», а реальные сэмплы). План «Б», если убивают: подсказка пользователю про батарейные настройки (dontkillmyapp-паттерн) + чекпоинт сессии в drift с восстановлением. |
| **R2** | iOS: приложение засыпает, если система сочтёт аудио «неиграющим» | низкая / высокий | Таймлайн — непрерывный PCM-поток (нулевые сэмплы ≠ остановка плеера); категория `.playback`; проверка в срезе №1 на реальном iPhone, вкл. режим энергосбережения. |
| **R3** | Методика измерения ±50 мс: без инструментального замера критерий §11.1 не доказуем | — | `tools/measure_drift.py` (детект атак в записи выхода) — делается в П5, ДО приёмки среза; loopback-кабель для чистоты замера. |
| **R4** | `flutter_tts.synthesizeToFile`: на Android — WAV, на iOS — .caf; формат/частота могут не совпасть с таймлайном; живой TTS нарушает критерий §11.2 (>200 мс) | средняя / средний | Декодер кэша нормализует PCM (caf-LPCM читается вручную, ресемплинг простым линейным интерполятором). Заглушка честно объявлена временной: финальное решение по ТЗ §7 — предзаписанные сэмплы (П13+/этап 2). Критерий §11.2 закрывается только предзаписью. |
| **R5** | Bluetooth-латентность (A2DP 100–300 мс): звук в наушниках отстаёт, вибрация на телефоне «спешит» относительно звука | средняя / низкий | Каналы аудио между собой синхронны всегда (один таймлайн). Для вибро: опция «сдвиг вибрации» или авто-отключение вибрации при BT (детект через audio_session devices). Открытый вопрос: читать ли `AudioManager.getOutputLatency` (Android) для автокомпенсации. |
| **R6** | Вибрация на iOS: пакет `vibration` не даёт тонких паттернов/амплитуд (Core Haptics не обёрнут) | высокая / средний | MVP: базовые паттерны (одиночная/двойная/длинная реализуемы через последовательность вызовов). Если UX слаб — мини-плагин на Core Haptics (этап 2, отдельная партия). |
| **R7** | Firebase vs Supabase — выбор зафиксирован (Firebase), но: vendor lock-in Firestore, цена при росте | низкая / средний | Соцмодель примитивна — слой `data/remote/` изолирует Firestore за интерфейсами репозиториев; миграция на Supabase — замена одного слоя. Push-логика (FCM) переносима: Supabase Edge Functions тоже шлют через FCM. |
| **R8** | Размер сборки ≤30 МБ (§11.5) | низкая / средний | Бюджет: Flutter arm64 ~12 МБ + аудио ≤5 МБ (наборы WAV + голоса ogg) + Rive ~0.1 МБ. Публикация app bundle / split-per-abi; контроль размера — шаг в П13; голосовые пакеты при росте — в asset-pack/on-demand. |
