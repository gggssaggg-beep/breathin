## Безопасность

- Ключи: в коде только publishable key (защита — RLS) — ок; Client secret нигде не встречается — ок.
- RLS: `sessions` — строго self (all) ✅; `challenge_participants` — select только со-участникам через security definer ✅, insert/update только своё ✅.
- `challenges` select `to authenticated using(true)` — **приемлемо**: нужно для вступления по коду, содержимое нечувствительно, код 31⁶; перебор упирается в rate limit Supabase.
- `profiles` select всем вошедшим — только display_name, приемлемо для v1 (имена и так видны со-участникам).
- Пробелы v1 (бэклог, не дыры): у создателя нет update/delete своего челленджа; участник не может выйти (нет delete-политики); М10 — лимит длины имени.
- Анонимный вход включён — мусорные аккаунты чистятся Supabase-джобом позже (Free-тариф, некритично).
